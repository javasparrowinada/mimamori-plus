Claude AIシステムの詳細

2026年4月30日
概要

このシステムが担う役割

Tuyaから取得したCSVを「家族が読みたい言葉」に変換するのが、Claude AIシステムの仕事です。

🎯 一言でいうと

「数字の羅列」を「暮らしの気配」に翻訳する仕組み。
ユーザーごとに使用パターンが違うため、固定テンプレートではなく AIが文脈を読んで文面を切り替えます。

月次バッチ 週次バッチ ユーザーごとに個別生成 パターン自動分類

⚙️ 各ステップの実装イメージ

1
CSV読込：Google Drive APIで「お客様情報.csv」「使用データ.csv」を取得。利用中ステータスのユーザーのみフィルタ。
2
パターン分析：Pythonで前処理（イベント数集計、曜日別集計、過去3ヶ月との差分計算）。AIに渡す前に「事実」を構造化。
3
文面生成：Claude APIに 構造化データ + プロンプトテンプレート を渡し、レポート文面を生成。
4
送信：SendGrid（メール）/ LINE Messaging API（LINE）/ Firebase（アプリ）へ振り分けて配信。
入力データ

AIに何を渡しているか

AIにCSVをそのまま投げるのではなく、Pythonで集計した「サマリーJSON」を渡します。
これによりトークン消費を抑え、出力の精度を上げます。

📥 AIに渡すJSON（例）
{ "user_id": "U-0012", "nickname": "お母さん", "month": "2026-04", "summary": { "active_days": 28, "total_days": 30, "light_days": 26, "usb_days": 22, "ac_days": 18, "sos_responses": "4/4", "outlet_connected_days": 30 }, "weekday_pattern": { "weekend_inactive_count": 2, "weekday_inactive_count": 0 }, "comparison": { "vs_last_month": {"active_days_diff": -1}, "trend_3months": [26, 29, 28] }, "pattern_type": "A" }

→ AIはこのJSONだけを見て、そのユーザー専用のレポート文を生成します。

分析ロジック

分析でやっていること

📊 5つの分析軸

ベースライン比較利用開始から最初の2週間で「そのユーザーの普通」を学習。以降の月次データと差分を取る。
曜日パターン「土曜は使用なし」など、繰り返される傾向を検出。突発的な無使用と区別する。
3ヶ月推移先月・先々月との比較で「安定している」「変化が見える」を判定。
パターン判定月間ライト点灯日数とイベント総数で A（よく使う） / B（あまり使わない） を自動分類。
文面バリエーション同じ内容でも毎月言い回しを変え、季節の話題を織り込む。マンネリ化を防ぐ。

🔀 パターン判定ルール

条件判定
月間ライト点灯日数 ≥ 20日
かつイベント総数 ≥ 50件		パターンA（よく使う）
月間ライト点灯日数 < 20日
またはイベント総数 < 50件		パターンB（あまり使わない）
コンセント未接続が48時間以上アラート対象（パターン判定の前に検知）

※ 閾値は運用しながらチューニング予定。

プロンプト設計

AIへの指示の組み立て方

Claude APIに渡すプロンプトは「役割定義 + ルール + データ + 出力形式」の4ブロック構成。

📝 プロンプト構造（抜粋）
# 役割 あなたは「見守りプラス」の月次レポート担当者です。 離れて暮らすご家族（見守る側）に、ご年配の親御さんの暮らしの気配を 伝える文面を書きます。 # 守るべきルール - 推定は推定として伝える（「〜かもしれません」） - ポジティブな解釈を先に置く - W数や専門用語は使わない - 「使っていない」を否定的に書かない - 季節の話題を1つ織り込む # 入力データ {summary_json} # 出力形式 - AIからのひとこと（120字以内） - 今月のトピック（3件まで） - 3ヶ月の推移コメント（60字以内）

🎨 文面生成の品質を保つ工夫

  • 固定テンプレート禁止：同じデータでも毎月違う表現になるよう温度パラメータを調整
  • ネガティブ語のブラックリスト：「使われていない」「異常」「心配」などは出力後にチェック
  • 長さ制限：トークン上限ではなく、文字数で制御してメール／アプリのレイアウトを崩さない
  • フォールバック文面：AI生成が失敗した場合に備え、テンプレ版を必ず用意

堅牢性

失敗したらどうするか

AI生成は確率的な処理なので、必ず「失敗しても止まらない」設計が必要です。

失敗ケース対応
Claude APIエラー / タイムアウト3回までリトライ。それでも失敗したらテンプレ版を送信し、運用者にSlack通知
Tuya API障害でCSV更新が止まる前回データを使って配信を継続。停止が24時間続いたら運用者に通知
AIが不適切な文面を生成（ネガティブ語含むなど）後段のフィルタで検知 → テンプレ版にフォールバック
メール / LINE 送信失敗キューに戻して翌時間に再試行。3回失敗で停止し通知
原則：「AIレポートが届かない」は絶対に避ける。
凝った文面が出せなくても、テンプレ版で「異常なし」を伝えるほうが見守りサービスとしては正しい。
プライバシー

AIに渡している情報

🔒 AIに渡す情報

  • ニックネーム（本名は渡さない）
  • イベント発生日時と種類
  • 集計値（月間使用日数、ライト点灯日数 等）

🚫 AIに渡さない情報

  • 本名・住所・電話番号
  • メールアドレス・LINE ID（送信処理側でのみ使用）
  • 映像・音声（そもそも取得していない）
方針：AIに渡すのは「暮らしの気配」を表現するために最低限必要な集計値のみ。
個人を特定できる情報は分離して扱います。
